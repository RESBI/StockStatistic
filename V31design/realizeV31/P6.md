# P6：HTTP/SSE 通讯、真实多进程与多 Worker 部署

## 1. 阶段目标

P6 把 P5 的 Embedded 完整链路映射到 [V3.1 通讯协议](../designV31/DESIGN_PROT_V31.md) 定义的 HTTPS/JSON、SSE 和 Artifact 数据面，实现真正独立进程、真实端口和多 Worker 运行，不再使用 TestClient 或后台线程冒充跨进程部署。

## 2. 必须范围

- Dispatcher FastAPI Gateway。
- Storage FastAPI control/data endpoints。
- SDK HTTP ControlChannel。
- SDK SSE EventChannel。
- SDK/Worker HTTP ArtifactChannel。
- Worker httpx 长连接 client。
- Client/Dispatcher/Storage/2+ Worker 独立进程 E2E。
- Docker Compose 开发栈。
- reconnect、timeout、Range、multipart。
- Cluster/capability query。
- 多 Worker scheduling 和 data locality 基础。

TLS/auth 可在开发环境简化，生产安全加固在 P9；协议和 scope hook 本阶段必须存在。

## 8. Docker Compose

开发栈至少：

```text
storage
dispatcher
worker-1
worker-2
```

本阶段可用 SQLite volume + LocalFS shared/object endpoint；同时提供 PostgreSQL/MinIO profile，为 P9 提前发现部署问题。

要求：

- health checks。
- 独立 volumes。
- 无源码目录隐式共享来传结果。
- Worker 只通过协议/Artifact endpoint 读取数据。
- 明确端口和环境配置。
- Ctrl+C/compose down 优雅停止。

## 9. 测试计划

### 9.1 Protocol conformance

- 所有 endpoint content type/version。
- golden ControlMessage HTTP round-trip。
- status code/error code。
- max control size 413。
- malformed/unknown major。

### 9.2 SSE

- 断线重连 after sequence。
- 事件重复去重。
- Dispatcher 重启后继续。
- partial ArtifactRef event。
- terminal 事件前 result 可读。

### 9.3 Artifact HTTP

- multipart upload。
- 50MB/500MB download streaming。
- Range resume。
- digest mismatch。
- SDK lazy materialization。
- Dispatcher 流量不承载 Artifact bytes 的网络断言/指标。

### 9.4 真实进程 E2E

必须用 subprocess 和 free ports：

- Client -> Dispatcher -> Storage -> Worker -> result。
- Indicator。
- Backtest。
- 两 Worker 并发多个 Jobs。
- Worker kill/retry。
- Dispatcher kill/restart。
- Storage 短暂不可用。
- Client 断开不取消 Job。
- Client 重连获取结果。

### 9.5 多 Worker

- 任务分布不超过 capacity。
- capability mismatch 不领取。
- priority 行为。
- 一个 Worker cache 热时 locality 优先。
- drain 后不领取新任务。

### 9.6 Docker smoke

- `docker compose up` 后 health ready。
- 从宿主 Client 提交 Job。
- 扩到 4 Worker。
- 重启任一 Worker/Dispatcher。
- volume 保持 result。

## 10. 性能与带宽

记录并门禁：

| 指标 | 目标 |
|---|---|
| Control submit P95 | < 100ms，不含 upload/snapshot |
| Claim P95 | < 50ms（本地网络） |
| SSE progress latency | < 1s |
| 50MB Snapshot 主库查询 | 1 次 |
| Dispatcher Artifact bytes | 接近 0，不代理大文件 |
| 32 slot 同数据 | Worker cache/object store，不重复 DB query |
| 多进程加速 | 可并行代表任务随 Worker/slot 增长有显著加速 |

不使用“4 个参数组合 <1s”作为分布式性能证明，应使用足够重的真实任务和串行基线。

## 13. 风险与处理

| 风险 | 处理 |
|---|---|
| TestClient 与真实网络差异 | P6 全部关键 E2E 使用 subprocess/ports |
| SSE proxy buffering | headers/heartbeat，Docker/Nginx profile 后续验证 |
| 大文件穿过 Python 内存 | streaming response/upload，memory benchmark |
| Worker 请求风暴 | long-poll、jitter、connection pooling |
| Windows subprocess flakiness | 明确 readiness probe、进程组清理 |

## 14. 退出条件

- HTTP/SSE/Artifact protocol conformance 通过。
- Client、Dispatcher、Storage、Worker 独立进程完整 E2E 通过。
- 至少两 Worker 的真实故障重试通过。
- Dispatcher 重启和 Client reconnect 不丢 Job。
- 大型 bytes 不穿 Dispatcher 控制面。
- Docker Compose 开发栈可重复启动、运行、停止。
