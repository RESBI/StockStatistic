# P1：旧行为基线、V3.1 工程骨架与 Contracts

## 1. 阶段目标

P1 在不改变现有 V3 运行代码的前提下完成三件事：

1. 冻结旧功能和金融数值基线。
2. 建立 V3.1 多包工程骨架和严格依赖边界。
3. 落地 `Job/Stage/WorkUnit/Attempt/Artifact` 等共享 Contracts 与协议 golden。

P1 不实现真实金融计算服务，不启动 Dispatcher/Storage/Worker 网络进程。

- [总体架构设计](../designV31/DESIGN_ARCH_V31.md)。
- [Foundation 架构设计](../designV31/DESIGN_ARCH_FOUNDATION_V31.md)。
- [通讯协议设计](../designV31/DESIGN_PROT_V31.md)。
- 当前 V3 全部测试和 PAXG 研究脚本。

## 3. 关键决策冻结

P1 结束前必须以 ADR 固定：

| ADR | 需冻结内容 |
|---|---|
| ADR-01 | V3.1 新包路径和发布名 |
| ADR-02 | Python 支持版本和 lock 策略 |
| ADR-03 | Pydantic/JSON schema 版本策略 |
| ADR-04 | UUIDv7、RFC 3339 UTC、半开区间 |
| ADR-05 | Canonical JSON 规则 |
| ADR-06 | ArtifactRef 最小/完整字段 |
| ADR-07 | capability ID/version 命名 |
| ADR-08 | 旧代码只作 oracle、禁止新包导入 |

若这些决策未冻结，不进入服务实现，避免后续所有包反复迁移 schema。

## 7. 测试计划

### 7.1 Contracts 单元测试

- 所有模型 valid fixture。
- 必填、类型、范围、unknown field invalid fixture。
- discriminated union 解析。
- datetime UTC normalization。
- 半开区间边界。
- ID round-trip 和排序。
- ErrorInfo nested causes。

### 7.2 Golden protocol

至少冻结：

```text
control_message.json
job_submit_indicator.json
job_submit_backtest.json
job_accepted.json
job_status_running.json
job_event.json
worker_register.json
work_lease.json
work_complete.json
error_stale_attempt.json
artifact_ref.json
dataset_snapshot.json
```

### 7.3 Property tests

- Canonical JSON 对 key 插入顺序不敏感。
- Canonical digest round-trip 稳定。
- 不同语义请求 digest 不相同。
- ArtifactRef tamper 导致 digest 校验失败。
- 时间分片组合无缺口/重叠。

### 7.4 Dependency tests

- 导入 Contracts 不加载 pandas/numpy/FastAPI/SQLAlchemy/httpx。
- Contracts 不能导入其他 V3.1 包。
- Dispatcher/Storage skeleton 不依赖 Kernel。
- 任意 V3.1 文件 import graph 不出现 legacy 根包。

### 7.5 Legacy baseline 校验

- Golden 生成脚本可重复运行。
- 连续两次固定 seed 生成一致摘要。
- fixture manifest 中 commit 和依赖版本完整。

## 10. 风险与处理

| 风险 | 处理 |
|---|---|
| 旧测试依赖机器/网络 | Golden 使用合成或冻结数据；网络测试单独记录 |
| 旧对象无法稳定序列化 | 提取规范字段到 Arrow/JSON，不保存对象 repr |
| Schema 过早膨胀 | Contracts 只放稳定原语，能力参数在 Kernel/能力模块 |
| Pydantic 默认行为漂移 | 固定 major/minor，golden 覆盖 serialization |
| Windows UUID/time 差异 | 使用纯逻辑测试和 UTC fixture |

## 11. 退出条件

- 新工程骨架可安装。

- 新代码未导入任何旧运行实现。
- 旧/新 package collision 由双环境测试证明已隔离。
