# P1：Contracts 与仓库骨架

> 前置：V3.1 设计文档已评审
> 输出：可独立安装的 `stockstat-contracts`、仓库工程基线和契约测试
> 后续：[P2 Storage](P2.md)、[P3 Finance Basic](P3.md)

## 1. 阶段目标

建立所有进程共同使用且不依赖重型计算或服务框架的契约层。P1 只解决“金融任务如何被稳定描述和验证”，不实现 Storage、Dispatcher、Worker 或金融算法。

阶段闭环：

```text
typed DTO -> canonical JSON -> digest -> schema validation -> state transition validation
```

## 2. 必需交付物

### 2.1 Monorepo 工程基线

创建首批真实目录：

```text
packages/contracts/
tests/contracts/
tests/architecture/
```

配置内容：

- Python 版本范围和统一构建后端。
- lint、format、type check 和 pytest 基线。
- 测试 marker：`unit`、`component`、`integration`、`parity`、`failure`、`performance`、`security`。
- Windows 和 Linux CI 基础矩阵。
- 构建 wheel 并从 wheel 安装测试，不依赖 editable install 掩盖包问题。

### 2.2 标识与时间

实现：

- `RequestId`、`TraceId`、`JobId`、`UnitId`、`AttemptId`、`LeaseId`、`WorkerId`、`SnapshotId`、`ArtifactId`。
- UUIDv7 生成和带前缀字符串校验。
- `sha256:<hex>` digest 类型。
- RFC 3339 UTC 解析和规范化。
- 明确拒绝 naive datetime、错误前缀、错误 digest 长度和大小写不规范。

### 2.3 Canonical JSON

实现唯一 canonical 编码函数，用于：

- request digest。
- Snapshot query digest。
- Artifact manifest digest。
- environment digest。
- operation parameter digest。

规则：

- key 字典序。
- UTF-8，无 BOM。
- 时间统一为 `Z`。
- 禁止 NaN/Infinity。
- decimal/货币使用明确字符串规范。
- 集合型字段必须在模型层定义顺序语义，编码器不擅自排序普通数组。
- 未声明字段默认拒绝。

### 2.4 核心资源模型

实现并导出 schema：

| 模型 | 最小内容 |
|---|---|
| `ArtifactRef` | id、kind、media type、schema、digest、size、location、created_at |
| `ArtifactMember` | name、ref、role/optional metadata |
| `ArtifactManifest` | version、kind、members、lineage summary |
| `SnapshotRef` | id、kind、schema、digest |
| `DatasetQuery` | instruments、timeframes、range、fields、revision、adjustment、timezone、quality |
| `DatasetPartition` | instrument、timeframe、ref、rows、min/max ts |
| `DatasetSnapshot` | query、partitions、lineage、digest |
| `CodeRef` | StrategyBundle ref、entrypoint、strategy API |
| `ResultManifestRef` | final/partial result reference |

模型只描述资源，不实现文件下载、pandas 转换或数据库读取。

### 2.5 Operation 契约

实现：

- `OperationId`，格式如 `backtest.run@1`。
- `SchemaRef` 和 operation-specific parameter/result schema 注册端口。
- `OperationDescriptor`。
- `ResourceRequirement`、`ResourceClass` 和 `SecurityProfile`。
- 确定性、可分片性、merge operation 和实现版本字段。

P1 提供最小测试 operation：

```text
test.echo_manifest@1
```

它只用于验证 schema/descriptor plumbing，不进入生产 operation discovery。

### 2.6 Job、WorkUnit 和 Worker DTO

实现：

- `JobSpec`、`JobPolicy`、`JobView`、`JobProgress`、`JobCounts`。
- `WorkUnitSpec`、dependency、partition、requirements 和 environment。
- `AttemptView`、`LeaseGrant`、`LeaseRenewal`。
- `WorkerRegistration`、capability manifest、resource pools、heartbeat。
- `ResultManifest` 和执行 metrics。
- 创建型请求所需的 idempotency metadata。

参数字段保持为受 schema 注册表校验的 JSON value，不在 `JobSpec` 堆叠回测、统计和搜索的可选字段。

### 2.7 状态与错误

实现有限状态机：

- Job：`accepted`、`planning`、`queued`、`running`、`cancelling`、`cancelled`、`succeeded`、`failed`。
- WorkUnit：`blocked`、`ready`、`leased`、`running`、`committing`、`succeeded`、`failed`、`cancelled`。
- Attempt：`created`、`started`、`succeeded`、`failed`、`lost`、`cancelled`、`stale`。
- Worker：`starting`、`online`、`busy`、`draining`、`offline`、`quarantined`。

实现 `ErrorInfo` 和 HTTP Problem schema，至少固定以下 category：

```text
validation, data, finance, code, resource,
execution, storage, protocol, security, internal
```

错误码与 `retryable` 默认值由显式表决定，不根据 message 文本推断。

### 2.8 事件与认证上下文

实现：

- `JobEvent`、单 Job `sequence`、progress 和 partial reference。
- `RequestContext`、tenant、subject、role/scope、request/trace ID。
- 控制面 DTO 中可传播的 W3C trace context 字段。
- ACL 只定义契约，不实现认证服务器。

### 2.9 Schema 发布

构建时输出版本化 JSON Schema：

```text
dist/schemas/common/
dist/schemas/market/
dist/schemas/jobs/
dist/schemas/workers/
dist/schemas/errors/
```

要求：

- schema ID 稳定。
- Python DTO 与 JSON Schema roundtrip。
- golden 文件变更可审查。
- 后续 OpenAPI 引用同一 schema 来源，不复制手写第二份定义。

## 3. 实现步骤

1. 建立包、测试和 CI 骨架，先验证 wheel 独立安装。
2. 实现 ID、时间、digest 和 canonical JSON。
3. 实现 Artifact/Snapshot 资源模型。
4. 实现 operation、Job、WorkUnit、Worker 和 event DTO。
5. 实现状态迁移表和错误分类表。
6. 生成 JSON Schema 和 golden。
7. 增加 architecture import tests，锁定 contracts 的依赖边界。
8. 发布 `0.1.0-dev` 内部构建，供 P2/P3 消费。

## 4. 测试计划

### 4.1 Canonical 与 digest

- 相同语义、不同 dict 插入顺序产生相同 bytes/digest。
- 数组顺序改变时 digest 改变。
- UTC offset 等价时间规范化为同一值。
- NaN、Infinity、naive datetime、非法 decimal 被拒绝。
- Unicode 使用 UTF-8 稳定编码。
- Windows/Linux golden bytes 相同。

### 4.2 Schema

- 每个 DTO 的最小合法 fixture。
- 缺失字段、未知字段、错误 enum、错误 ID 前缀。
- operation 参数 schema 独立校验。
- Python -> JSON -> Python roundtrip。
- JSON Schema validator 与 Python validator 对同一 fixture 结论一致。

### 4.3 状态机

- 每条允许迁移均有测试。
- terminal state 不可再次迁移。
- `state_version` 必须单调递增。
- `succeeded` 必须带 final result ref。
- `cancelled` 不得带成功结果。
- `committing` 不等于成功。

### 4.4 错误与事件

- 错误 category、code 和默认 retryable 映射。
- traceback 不能作为普通 `detail` 的必填内容。
- Job event sequence 和 event payload schema。
- progress fraction 边界为 0..1。
- partial event 只允许 ArtifactRef，不允许内联 table。

### 4.5 架构边界

- contracts wheel 不依赖 pandas/numpy/scipy/FastAPI/SQLAlchemy/httpx/cloudpickle。
- contracts 模块 import 不触发网络、文件系统或数据库访问。
- 包安装后 schema 资源可发现。

## 5. 验收门禁

P1 完成必须满足：

1. `stockstat-contracts` wheel 可独立安装和 import。
2. 所有核心 DTO 有 JSON Schema golden。
3. canonical/digest golden 在 Windows 和 Linux 一致。
4. 状态机合法/非法迁移测试全部通过。
5. contracts 依赖扫描未发现禁止依赖。
6. 一个合法 `JobSpec` 可以验证、canonical 编码、计算 digest 并 roundtrip。
7. 一个带 SnapshotRef 的 `WorkUnitSpec` 和 committed ResultManifest 可以完整 roundtrip。
8. 不存在 V2/V3 `TaskSpec`、`ComputeSpec`、`Envelope` 或兼容 shim。

## 6. 非目标

- 不启动 HTTP 服务。
- 不创建数据库表。
- 不实现真实 operation executor。
- 不引入 pandas/Arrow 数据容器。
- 不实现 SDK facade。
- 不冻结尚未被 P2-P4 消费的所有未来金融 schema。

## 7. 风险与处理

| 风险 | 处理 |
|---|---|
| DTO 过早膨胀 | 只定义跨进程稳定资源；业务字段留在 operation schema |
| schema 库泄漏到所有 API | 对外导出稳定 DTO 和 schema，不暴露库内部 validator 类型 |
| canonical float 跨平台差异 | 控制面避免计算型浮点；固定编码规则并使用 golden |
| P1 追求一次冻结全部契约 | 仅冻结 P2-P4 必需模型，P4 前允许受控调整 |
| ID 类型增加使用成本 | SDK 层后续提供字符串便利，contracts 保持严格 |
