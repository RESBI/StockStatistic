# P9：生产硬化、性能基线与正式切换

> 前置：[P8 User Surface and Migration](P8.md)
> 输出：可发布 V3.1、生产部署基线和旧 runtime 移除

## 1. 阶段目标

把功能完整的 V3.1 候选系统提升到可正式切换状态。P9 聚焦安全、故障恢复、性能、可观测性、数据保留、发布和删除旧 runtime，不再引入新的大型金融功能。

阶段闭环：

```text
release candidate -> failure/security/performance/deployment gates
-> migration rehearsal -> V3.1 default -> remove legacy runtime
```

## 2. 必需交付物

### 2.1 认证、授权和多租户

完成生产端口和至少一个参考实现：

- Client Bearer/OIDC validation。
- Worker mTLS 或独立 worker credential。
- tenant、subject、role、scope。
- Job、Snapshot、Artifact、StrategyBundle ACL。
- scoped/short-lived download、upload 和 lease token。
- key/token rotation 流程。
- 认证失败不泄露资源存在性。
- audit event。

最小 scopes：

```text
jobs:submit, jobs:read, jobs:cancel,
market:read, market:write,
artifacts:read, artifacts:write,
cluster:read, cluster:manage
```

### 2.2 Strategy Worker sandbox

生产 Linux 门禁：

- trusted Worker 和 Strategy Worker 独立 pool/image/service account。
- non-root、read-only root filesystem、独立 workspace。
- network default deny，只允许必要 Storage/Dispatcher control path。
- CPU/memory/pid/temp disk limits。
- seccomp/AppArmor 等部署策略。
- bundle signature/trusted issuer allowlist。
- dependency lock/image compatibility。
- no runtime public package install。
- traceback/diagnostic Artifact ACL。

Windows 本地模式只承诺开发级隔离，文档必须明确不能等同生产 sandbox。

### 2.3 配额与滥用防护

实现：

- Job submit rate/active count。
- DAG unit/edge 上限。
- grid/random trial 上限。
- Monte Carlo sample 上限。
- Artifact/request/upload size。
- per-tenant storage/retention。
- per-tenant active slots/CPU budget。
- SSE connections/event rate。
- bundle expanded size/file count。
- admin override 和审计。

### 2.4 数据完整性、保留和灾备

完成：

- PostgreSQL backup/restore rehearsal。
- blob versioning/backup policy。
- metadata/blob consistency checker。
- orphan upload/blob cleanup。
- manifest graph GC。
- retention/pin/legal hold。
- checksum audit 和 corruption detection。
- disaster recovery runbook，定义 RPO/RTO。
- PAXG research Snapshot/result pin。

任何 GC 都先支持 dry-run 和可审计报告。

### 2.5 Dispatcher/Worker 可靠性

强化：

- multi-replica stateless Dispatcher API。
- scheduler leader/DB coordination。
- no duplicate valid lease under failover。
- lease expiry/retry sweeper health。
- Worker drain/rolling upgrade。
- protocol/kernel/capability version compatibility policy。
- quarantine and recovery。
- graceful shutdown deadline。
- stuck Job/unit reconciliation tool。

仍不实现多级 Dispatcher。

### 2.6 可观测性和 SLO

完善 dashboards/alerts：

- API availability/latency/error rate。
- submit/planning latency。
- queue age by resource class。
- execution duration/error/retry by operation。
- active/expired/stale leases。
- Worker online/slot/cache/resource。
- Storage request/blob/DB latency and errors。
- Artifact upload/commit failures。
- SSE reconnect/cursor expiry。
- tenant quota/rejection。
- lineage/GC integrity。

定义初始 SLO 和 error budget。数值根据 P5-P8 实测基线确定，不在设计阶段伪造极限指标。

### 2.7 性能基准

建立可重复 benchmark harness 和环境记录：

#### Control plane

- Job create/status/list/event throughput。
- lease acquire/renew/complete transaction latency。
- 1k/10k unit DAG planning/persistence 的受控测试。
- Dispatcher RSS 与 Snapshot/result bytes 解耦。

#### Data plane

- 50 MB、500 MB、PAXG Snapshot 读取。
- Artifact upload/commit/download。
- local file locator vs HTTPS/S3。
- cold/warm Worker cache。
- Parquet predicate/column projection。

#### Finance

- PAXG P3 全研究链。
- 单 PAXG backtest。
- 208-run batch with 1/2/4/8 Workers。
- 3 benchmark comparison 和 Bayesian study resume/reference mode。
- 1k/10k Monte Carlo samples。
- spectral/nonlinear operation。

验收重点：

- 无 V3 base64/cloudpickle 结构性放大。
- 多 Worker 有可解释吞吐提升。
- Dispatcher 不成为数据带宽瓶颈。
- cache warm run 明显减少重复下载。
- 无失控进程/BLAS 过度订阅。

### 2.8 全量故障演练

自动化或半自动化场景：

- Dispatcher kill/restart/failover。
- Worker Agent/child process kill。
- Storage API restart。
- PostgreSQL/MinIO short outage。
- network latency/drop/partition。
- result commit 后 complete 前断线。
- complete 后 response 丢失。
- cancel/complete/lease expiry race。
- disk full、OOM、temp disk exhaustion。
- corrupted download/cache/blob。
- expired token/certificate rotation。
- rolling version upgrade。

每个场景定义 expected state、retry behavior、operator signal 和 cleanup。

### 2.9 发布工程

完成：

- packages/services semantic versions。
- reproducible wheel/container builds。
- dependency lock/SBOM。
- vulnerability/license scan。
- container signing/provenance。
- migration compatibility matrix。
- release notes 和 known limitations。
- rollback plan，只回滚 V3.1 release，不恢复双 runtime 写入。
- upgrade/migration tests from first supported V3.1 prerelease schema。

### 2.10 数据和代码迁移

实现一次性工具：

- 旧 SQLite/Parquet OHLCV importer -> source revision/Snapshot。
- 本地 research file importer。
- 旧配置/profile conversion report。
- Strategy migration scanner/report。
- Artifact export/import 用于环境搬迁。

不导入 V3 Job 内存状态、cloudpickle 结果或运行中任务。V3 Job ID 只留历史记录，不作为 V3.1 可继续执行资源。

### 2.11 切换与删除旧 runtime

切换步骤：

1. 冻结旧 runtime 新功能。
2. 运行完整 migration rehearsal 和 PAXG release candidate。
3. 将根 README、默认安装、CLI 和 compose 指向 V3.1。
4. 发布 V3.1 并监控 canary 环境。
5. 停止旧 Dispatcher/Worker 接受新任务。
6. 导出需要保留的数据和历史报告。
7. 删除旧 runtime 代码、兼容层、旧服务入口和不再适用测试。
8. 保留 legacy behavior fixtures、必要研究输出和历史设计文档。
9. 再次运行 import boundary、docs、fresh install 和 full parity。

删除范围应通过单独审查清单确认，避免误删研究 fixture 或用户数据。

## 3. Release Gate 测试

### 3.1 功能与 parity

- 当前功能 migration matrix 全绿。
- PAXG v1-v7 研究 parity。
- 52 x 4 = 208 backtest runs + 3 benchmarks。
- search/MC/walk-forward/shard parity。
- local/remote/multi-worker parity。
- render/report and lineage。

### 3.2 安全

- tenant isolation。
- scope/role matrix。
- token/lease/upload replay。
- signed/unsigned/tampered StrategyBundle。
- zip slip/symlink/zip bomb。
- path traversal/file locator。
- SSRF/no arbitrary URL result refs。
- upload/media/schema/size limits。
- strategy network/filesystem/resource restrictions。
- dependency and image scans。

### 3.3 可靠性

- failure matrix 全部有预期 terminal/recovery state。
- Dispatcher restart/failover 不丢任务。
- Worker loss safely retries。
- duplicate/stale complete 不覆盖结果。
- Storage commit atomicity。
- backup restore 后 pinned Snapshot/results 可用。
- reconciliation/GC 不删除可达资产。

### 3.4 性能

- benchmark 环境和 raw data 可复现。
- Dispatcher RSS/body instrumentation 证明数据不经过控制面。
- cold/warm cache report。
- 1/2/4/8 Worker throughput/efficiency report。
- PAXG 208-run completion baseline。
- queue/lease/DB saturation point 和 capacity guide。

### 3.5 部署

- local Windows quickstart。
- local Linux quickstart。
- compose single Worker。
- compose N Worker/pool split。
- PostgreSQL/MinIO production-like setup。
- rolling upgrade/drain。
- fresh install and backup restore。

### 3.6 文档与迁移

- all executable examples。
- link/schema/OpenAPI checks。
- migration scanner corpus。
- legacy import scan of V3.1 source = 0。
- no default docs reference old commands。
- unsupported/removed behavior clearly listed。

## 4. 验收门禁

V3.1 只有同时满足以下条件才能正式切换：

1. P1-P8 阶段门禁仍全部通过。
2. 当前功能矩阵 100% 有 V3.1 入口或明确批准的移除项。
3. PAXG v1-v7、307 样本、52 native strategies、208 runs、3 benchmarks 和复合实验全部通过。
4. local/remote/N Worker、Windows/Linux 和部署测试通过。
5. Dispatcher restart、Worker loss、Storage/DB 短暂失败和竞态测试通过。
6. 大数据不经过 Dispatcher 的结构和性能证据通过。
7. 多租户 ACL、Worker credential 和 Strategy sandbox 安全门禁通过。
8. backup/restore、retention、GC 和 corruption 检查通过。
9. release artifacts 可复现、带 SBOM/签名并通过扫描。
10. migration guide/scanner/importer 和 rollback runbook 完成演练。
11. 根文档和默认部署只使用 V3.1。
12. 旧 runtime 删除后 fresh install、full tests 和 PAXG release suite 再次通过。

## 5. 切换后监控

首个正式版本设观察期：

- error budget 和 SLO 每日审查。
- stale lease/retry/failed Job 趋势。
- Artifact/Storage growth 和 GC dry-run。
- Worker resource/OOM/cache hit。
- tenant quota/rejection。
- migration issue 分类，不通过恢复旧 runtime 解决。
- 任何金融结果差异必须关联 Snapshot、strategy、operation 和 environment digest 调查。

## 6. 非目标

- 不因发布硬化引入多级 Dispatcher。
- 不新增任意函数执行或通用工作流。
- 不在切换阶段扩展 ML、期权、订单簿等新领域。
- 不承诺 Windows 本地模式具有 Linux 生产 sandbox 等级。
- 不把旧 runtime 作为永久 fallback 共同维护。
- 不通过放宽测试容差掩盖 release regression。

## 7. 风险与处理

| 风险 | 处理 |
|---|---|
| 硬化阶段不断加入新功能 | feature freeze，P9 只修 release blocker |
| 删除旧代码误删 fixture/数据 | 删除清单、备份、分步审查和 full parity rerun |
| 性能目标脱离硬件 | 记录环境和相对基线，发布 capacity guide |
| sandbox 在 Windows 不等价 | 生产 Linux 强门禁，Windows 明确开发级限制 |
| migration 失败导致双 runtime 长期共存 | rehearsal、importer、canary 和明确停止旧服务日期 |
| GC/retention 造成不可逆丢失 | dry-run、pin、backup restore 和审计后才启用删除 |
