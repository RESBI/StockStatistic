# P9：生产加固、HA、性能安全验收与一次性切换

## 1. 阶段目标

P9 将已具备功能完整性的 V3.1 加固为可发布架构，完成 PostgreSQL/Redis/S3-compatible 生产 adapter、Dispatcher 多副本、认证授权、可观测、容量/故障测试、部署运维文档，并执行公共包和仓库的一次性切换。

P9 不是继续增加金融功能；任何非必要新能力应推迟，优先证明可靠、安全、可运维。

## 7. Autoscaling

输出标准 metrics/hook，不把 K8s API 写入 Dispatcher 核心：

- queued resource demand。
- oldest queue wait。
- available capability-specific capacity。
- active/leased work。
- recent task duration。
- safe scale-down candidates。

测试：

- scale-up metric 在持续 backlog 触发。
- drain 后现有 Attempt 完成。
- 不 scale-down 最后必要 capability Worker。
- cache locality/大数据任务避免频繁抖动。

## 8. 故障与混沌测试

强制场景：

| 故障 | 验收 |
|---|---|
| Worker Agent kill | Lease 到期、retry、Job 完成 |
| Executor kill/OOM | Agent 存活、错误分类、按策略 retry/terminal |
| Dispatcher replica kill | Client/Worker 重连，Job 不丢 |
| 全部 Dispatcher rolling restart | 状态恢复、lease 可续或安全到期 |
| Redis flush/down | 从 PG 重建，无 Job 丢失 |
| PostgreSQL 短中断 | 服务退避、无状态破坏，恢复后继续 |
| MinIO/S3 短中断 | upload/download retry，digest 不变 |
| Client disconnect | Job 继续 |
| SSE proxy disconnect | Last-Event-ID 续读 |
| Storage snapshot 中断 | 无半成品可见，可重试 |
| network partition 导致双 Attempt | fencing 只接受当前 Attempt |
| disk full | 明确错误、cache eviction、无 silent corruption |

## 9. 性能与容量测试

### 9.1 Micro

- submit/status/claim/renew/complete P50/P95/P99。
- SSE fan-out。
- Artifact upload/download throughput。
- Snapshot Arrow encode。
- Task Store query/index。

### 9.2 Macro

- PAXG 132 回测多 slot。
- 1000/10000 trial search。
- Monte Carlo 10000+。
- 50MB/500MB/3GB 数据快照和多 Worker。
- 100 Worker 模拟 heartbeat/claim；若无真实机器，用进程/协议负载器，但至少保留真实 Worker 子集。
- 10 万 WorkUnit Job 状态与事件分页。

### 9.3 门禁

最终门限根据 P6/P7 基线固定，至少包括：

- 无线性增长的 Dispatcher RSS 随结果大小。
- Storage 主库查询不随 WorkUnit 数增长。
- Worker slots 有有效加速。
- stale/duplicate 不改变结果。
- Event/DB/Artifact 保留策略下容量可预测。

## 12. 切换计划

### 12.1 切换前门禁

- P1-P8 全部测试通过。
- PAXG/现有功能迁移报告通过。
- 生产 adapter/HA/security/chaos/perf 通过。
- README/USAGE/发布说明完成。
- 旧 API 不兼容清单和迁移工具发布。
- 数据库迁移 dry-run 在真实副本执行。

### 12.2 发布顺序

1. 发布 `stockstat-contracts` 和 `stockstat-kernel`。
2. 发布 Storage/Dispatcher/Worker images。
3. 部署独立 V3.1 stack，运行 shadow validation。
4. 发布新 `stockstat` SDK 和 migration CLI。
5. 用户脚本迁移/验证。
6. 设定切换窗口，停止旧写入或冻结旧数据源。
7. 最终增量数据迁移。
8. 切换 endpoint/package 默认。
9. 运行 post-cutover smoke/parity。
10. 保留旧系统只读短期回查，不双写长期运行。

旧、新 Python distribution 使用同一 `stockstat` import 名。Shadow validation 必须由 `.venv-legacy` 与 `.venv-v31` 两个独立子进程执行；正式切换时卸载旧 distribution 后再安装新 SDK，不允许依赖安装顺序覆盖同名模块。

### 12.3 代码切换

切换完成后：

- 新 `packages/sdk` 成为顶层 `stockstat`。
- README/默认命令仅描述 V3.1。
- 旧 `frontend/backend/worker` 移入明确 `legacy/` 或在确认无需后删除。
- 不让新 SDK import legacy。
- 旧 V3 测试归档为 migration fixtures，不继续作为新代码结构门禁。
- 设计文档状态更新为实际实现，且数字来自 CI 报告。

### 12.4 回滚

回滚只针对部署路由和新写入窗口，不尝试让新 Job 在旧 V3 中继续执行：

- 切换前保留旧只读/可启动镜像。
- 数据迁移有备份。
- V3.1 Job/Artifact schema 不写入旧数据库。
- 如回滚，停止接受新 V3.1 Job，等待/取消当前 Job，恢复旧 endpoint。
- 已完成 V3.1 Result 通过 Artifact 导出保留。

## 14. Go/No-Go 清单

### 14.1 Go

- [ ] 所有金融功能迁移矩阵为绿色。
- [ ] 无 P0/P1 级数据损坏或权限漏洞。
- [ ] 可靠性不变量全部自动测试。
- [ ] 生产备份/恢复演练成功。
- [ ] 观测和告警可用。
- [ ] 迁移文档与工具经过真实脚本验证。
- [ ] 性能满足门限。
- [ ] 运维负责人批准 runbook。

### 14.2 No-Go

以下任一存在即不切换：

- stale Attempt 可覆盖新结果。
- Dispatcher/Redis 重启会丢 Job。
- Artifact digest/权限不可验证。
- 远程策略仍依赖 pickle。
- PAXG/关键回测有未解释结果漂移。
- Worker crash 需要人工修复任务。
- 大数据仍经 Dispatcher base64 中转。
- 生产测试因服务缺失被跳过。

## 15. 退出条件

- V3.1 生产拓扑、HA、安全、可观测和运维全部可用。
- 全量测试/chaos/perf/security 通过。
- 数据和客户代码迁移演练成功。
- 新公共包和默认文档完成切换。
- 新实现无 legacy runtime import。
- 旧系统已进入明确只读/归档/删除流程，不长期双轨。
