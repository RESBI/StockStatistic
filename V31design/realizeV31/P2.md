# P2：Storage 数据、快照与资产

> 前置：[P1 Contracts](P1.md)
> 输出：本地可运行的 Storage 服务与 Python service adapter
> 后续：[P4 Durable Local Jobs](P4.md)

## 1. 阶段目标

实现市场数据 revision、不可变 DatasetSnapshot、Artifact 两阶段提交和 lineage。P2 形成独立数据闭环，不依赖 Dispatcher 或 Finance Kernel。

阶段闭环：

```text
source/import -> normalized revision -> snapshot -> download
worker-style upload -> commit manifest -> download -> lineage
```

## 2. 首期技术范围

| 领域 | 本地实现 | 集成验证 |
|---|---|---|
| metadata | SQLite | PostgreSQL |
| blob | content-addressed filesystem | MinIO/S3 compatible |
| table format | Parquet + ZSTD / Arrow IPC | 相同 |
| API | InProcess service + HTTP | HTTP |
| ingest | Synthetic、file importer、Binance、Yahoo、Coinbase | fake source + 可选真实 smoke |

外部数据源测试默认使用录制或 fake adapter，CI 不依赖公网稳定性。

## 3. 必需交付物

### 3.1 Storage 包和端口

创建：

```text
services/storage/
tests/services/storage/
tests/integration/storage/
```

核心端口：

- `MetadataRepository`。
- `BlobStore`。
- `SourceAdapter`。
- `StorageService`。
- `Clock` 和 transaction boundary。

HTTP 路由和 InProcess adapter 调用同一 application service，禁止复制两套业务逻辑。

### 3.2 元数据模型与迁移

建立并迁移：

- instruments。
- ingest requests。
- source revisions。
- normalized partitions。
- snapshots 和 snapshot partitions。
- artifacts、artifact blobs、artifact members。
- upload sessions。
- lineage edges。
- retention/pins/ACL。
- idempotency keys。

所有 immutable 资源禁止 UPDATE 内容字段；允许更新的只有 retention、pin、访问时间和管理状态。

### 3.3 Instrument 与 MarketTable schema

实现：

- `InstrumentId` catalog 和 native symbol 映射。
- OHLCV Arrow schema：UTC timestamp、instrument、timeframe、OHLCV、source revision、quality flags。
- timeframe、字段、排序和唯一键校验。
- UTC 转换和明确的交易所本地时区 metadata。
- Parquet 分区写入与读取。

### 3.4 Ingest 和 revision

优先顺序：

1. Synthetic adapter，用于确定性测试。
2. Parquet/CSV importer，用于导入现有 PAXG fixture 和旧 SQLite 导出。
3. Binance/Coinbase ccxt adapters。
4. Yahoo Finance adapter。

流程要求：

- 每次采集生成 ingest resource。
- fetch、normalize、quality、write partition、commit revision 分阶段记录。
- 增量采集基于已提交 revision coverage。
- 回补产生新 revision，不覆盖旧数据。
- 同 idempotency key 同请求返回已有资源；不同请求冲突。

### 3.5 数据质量

实现首批策略：

- `research_strict@1`。
- `research_warn@1`。
- `ingest_permissive@1`。

检查：

- schema/type。
- instrument/timeframe/ts 唯一性。
- 时间单调。
- coverage 和缺失 bar。
- `low <= open/close <= high`。
- 非负 volume。
- timezone 为 UTC。
- outlier flag，不默认删除。

质量报告作为 Artifact 或 manifest 成员保存。

### 3.6 Snapshot

实现 `DatasetQuery -> DatasetSnapshot`：

- 解析 instrument、timeframe、范围和 revision policy。
- 覆盖检查。
- canonical query + revision digest。
- 相同输入幂等复用。
- manifest-only snapshot。
- 可选 materialized snapshot，为后续性能优化保留但不是默认。
- partition metadata、schema、quality 和 lineage。

Snapshot 提交后成员不可改变。新增数据只能产生新 Snapshot。

### 3.7 Artifact 数据面

实现：

- 创建 upload session。
- 声明 name、kind、media type、schema、size 和 digest。
- filesystem direct upload 和 HTTP PUT target。
- commit 时校验 bytes、digest、size、schema 和 owner scope。
- 原子发布 manifest。
- metadata 查询、成员列表和下载 locator。
- 相同 blob digest 去重，但保留独立 artifact lineage/ACL。
- 未 commit upload 不可见。

### 3.8 Lineage 与保留

每个 Snapshot/Artifact 记录 parent、operation、parameters digest、job/unit/attempt、环境和创建主体。实现：

- 单资源 lineage 查询。
- parent/child 遍历的安全深度上限。
- pin/unpin。
- dry-run GC。
- 默认 retention policy。
- 清理 expired upload session 和 orphan blob。

### 3.9 HTTP 资源

实现设计协议的最小子集：

```text
GET/POST /v31/instruments
POST     /v31/ingests
GET      /v31/ingests/{id}
GET      /v31/revisions/{id}
POST     /v31/snapshots
GET      /v31/snapshots/{id}
GET      /v31/artifacts/{id}
POST     /v31/artifacts/{id}/download
POST     /v31/upload-sessions
POST     /v31/upload-sessions/{id}/commit
GET      /v31/lineage/{id}
```

HTTP Problem、request ID、protocol header 和 idempotency 复用 contracts 定义。

## 4. PAXG 数据迁移 fixture

使用现有真实文件建立固定 importer 测试：

```text
working/PAXG-Weekend-Monday-Law-v5-redo/results/paxg_1d.parquet
working/PAXG-Weekend-Monday-Law-v5-redo/results/paxg_1h.parquet
working/PAXG-Weekend-Monday-Law-v5-redo/results/btc_1d.parquet
```

测试前记录源文件 SHA-256。导入后应得到：

- PAXG/USDT 1d revision。
- PAXG/USDT 1h revision。
- BTC/USDT 1d revision。
- 一个覆盖 2020-08-28 至 2026-07-16 的 immutable Snapshot。
- 行数、时间范围、schema 和 digest 报告。

若真实 fixture 未来更新，必须创建新 golden 版本，不能覆盖原 digest。

## 6. 测试计划

### 6.1 功能

- Synthetic 全量与增量采集。
- file importer 的 schema 映射和 UTC 转换。
- Binance/Yahoo/Coinbase adapter 的分页、限流和错误映射 fixture。
- revision 不覆盖。
- Snapshot 单/多标的、单/多 timeframe。
- manifest-only 和 materialized 内容一致。
- Artifact 单成员和多成员 manifest。
- selected member download。
- lineage 完整性和 ACL scope。

### 6.2 幂等与并发

- 并发同 ingest key 只执行一次。
- 并发相同 Snapshot 请求返回同一 digest/resource。
- 并发相同 blob 上传安全去重。
- 同 upload commit 重发返回 duplicate。
- 同 key 不同 request digest 返回冲突。

### 6.3 故障

- blob 写完、metadata commit 前崩溃。
- metadata 写入失败后不暴露 Snapshot。
- 上传中断、digest 错误、size 错误。
- commit 后响应丢失并重发。
- SQLite/PostgreSQL 重启后资源可读。
- MinIO 短暂不可达后安全重试。
- expired signed URL 和 file locator 越权。

### 6.4 格式与数值

- Arrow/Parquet roundtrip 不改变 OHLCV 数值和 UTC。
- partition 合并顺序稳定。
- predicate range 读取边界明确为 `[start, end)`。
- JSON 不返回大表或 base64。

### 6.5 性能基线

- 50 MB 和 500 MB Artifact 上传/下载不进入 API 进程大对象 JSON。
- Snapshot manifest 查询延迟与数据大小近似无关。
- 相同 Parquet blob 只存一份物理内容。
- PAXG 1h fixture 可在约定开发机资源内导入和读取。

## 7. 验收门禁

1. SQLite + filesystem 本地 Storage 可从空目录启动和迁移。
2. PostgreSQL + MinIO 集成套件通过。
3. 四数据源 adapter 均有离线 fixture 测试，Synthetic 和 importer 可实际运行。
4. 相同 query + revisions 产生相同 Snapshot digest。
5. upload 只有 commit 后可见，重复 commit 幂等。
6. PAXG/BTC fixture 成功导入并生成固定 Snapshot golden。
7. Storage 重启后 Snapshot、Artifact 和 lineage 均可读。
8. HTTP 和 InProcess 对同请求返回等价 DTO/错误码。
9. Storage 未 import Finance Kernel、Dispatcher 或旧 runtime。

## 8. 非目标

- 不调度 ingest Worker。
- 不执行指标、统计或回测。
- 不维护 Job 状态。
- 不提供任意 SQL API。
- 不完成复杂 S3 multipart 优化或跨区域复制。
- 不立即删除旧 SQLite 数据；只提供后续 importer 所需端口。

## 9. 风险与处理

| 风险 | 处理 |
|---|---|
| revision 和 Snapshot 语义混淆 | revision 表示源更新，Snapshot 表示下游不可变选择 |
| SQLite/PostgreSQL 行为差异 | repository contract suite 对两个后端运行 |
| 文件路径泄露 | 对外只返回逻辑 ref 或 scoped locator |
| ingest 网络测试不稳定 | CI 使用 fake/recorded adapter，真实 smoke 非阻塞 |
| GC 删除共享资产 | manifest 图、retention、pin 和 dry-run 先行 |
