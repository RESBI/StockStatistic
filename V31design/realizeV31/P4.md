# P4：Durable Local Jobs 与统一 Local Session

> 前置：[P1 Contracts](P1.md)、[P2 Storage](P2.md)、[P3 Finance Basic](P3.md)
> 输出：持久 Dispatcher、LocalWorker 子进程和首个统一 SDK 闭环
> 后续：[P5 Remote Worker](P5.md)

## 1. 阶段目标

完成第一条真实端到端执行链。用户通过 `StockStat.local()` 提交类型化 Job，Dispatcher 将状态持久化到 SQLite，LocalWorker 通过 lease 获得 WorkUnit，在子进程执行 P3 capability，先把结果提交 Storage，再完成 Job。

阶段闭环：

```text
SDK -> InProcess Job API -> SQLite JobStore -> lease -> child process
    -> Local Storage commit -> complete -> JobResult
```

本地模式必须使用真实状态机、租约、序列化边界和结果资产，不能退化为 SDK 直接调用 Kernel。

## 2. 必需交付物

### 2.1 Dispatcher 包和 JobStore

创建：

```text
services/dispatcher/
services/worker/
packages/sdk/
tests/services/dispatcher/
tests/services/worker/
tests/e2e/local/
```

SQLite JobStore 表：

- jobs。
- work_units。
- attempts。
- leases。
- workers/sessions。
- job_events。
- idempotency_keys。

实现 migration、transaction repository 和 crash-safe recovery。

### 2.2 Operation catalog 和验证

Dispatcher 启动时加载 P3 descriptor/planner metadata：

- operation discovery。
- JobSpec 顶层和 parameter schema 校验。
- input kind/schema 校验。
- resource/security requirements。
- implementation compatibility。

Dispatcher 不 import pandas/scipy；必要时把 planner metadata 分成轻量子包。

### 2.3 首批 Planner

P4 只实现单元计划和最小依赖：

- 单 execute unit。
- 可选 finalizer unit，仅用于验证依赖和 reducer 机制。
- DAG cycle/size 验证。

首批可提交 operation 使用 P3 清单。Batch/grid/Monte Carlo/walk-forward 的正式 planner 延至 P7。

### 2.4 Lease 与 attempt

实现：

- Worker 注册和 session replacement。
- acquire with capability/resource pool。
- 单事务创建 attempt、lease 和状态迁移。
- start、renew、progress、complete、fail、cancelled。
- lease expiry scanner。
- stale lease 拒绝。
- complete `(attempt_id, manifest_digest)` 幂等。
- retry policy、`not_before` 和有限指数 backoff。
- attempt exhaustion 后 unit/job failed。

SQLite 并发模型要有明确限制；本地只需要单 Dispatcher，但必须正确处理 Worker agent 与 API 的并发事务。

### 2.5 LocalWorker Agent

实现最小 Worker：

- capability scan 和 self-test。
- `spawn` 子进程执行，Windows/Linux 一致。
- slot manager，默认 `cpu-small=1`，可配置多 slot。
- task workspace。
- 从 Local Storage 解析 Snapshot/Artifact。
- digest cache。
- progress 节流和 lease renew。
- result upload/commit 后 complete。
- cooperative cancellation + grace + terminate。
- 子进程 exit/timeout/error 分类和诊断 Artifact。

### 2.6 Local runtime composition

`StockStat.local(runtime_dir)` 启动或组合：

- SQLite Storage metadata。
- filesystem blob store。
- SQLite JobStore。
- InProcess Dispatcher/Storage adapters。
- LocalWorker agent 和任务子进程。

生命周期要求：

- 明确 `close()` 和 context manager。
- runtime lock，避免两个进程误用同一 SQLite 本地目录。
- 第二次打开同 runtime 可恢复现有 Job 和资产。
- 初始化和 migration 幂等。

### 2.7 最小 SDK

实现：

```python
StockStat.local(...)
session.market.from_pandas(...)
session.market.snapshot(...)
session.features.indicators(...)
session.statistics.describe(...)
session.statistics.hypothesis(...)
JobHandle.status/wait/cancel/events/result
JobResult.as_table/lineage
```

同步 helper 必须内部 submit + wait。DataFrame 先提交临时 Artifact/Snapshot，不进入 Job JSON。

### 2.8 事件流

本地实现可恢复的事件迭代接口：

- Job event sequence 持久化。
- after sequence 读取。
- terminal event 后结束。
- progress/partial 只携带小型 metadata/ref。
- Subscriber 断开不影响任务。

P5 将同一接口暴露为 SSE。

### 2.9 结果终结规则

Job 成功必须满足：

1. 当前 attempt lease 有效。
2. ResultManifest 已 committed 且 owner scope 匹配。
3. manifest digest 与 complete 请求一致。
4. unit 原子转 succeeded。
5. 依赖 unit ready 或 final Job result 写入。
6. `job.succeeded` event 与 Job terminal state 同事务。

3. 实现 Worker registry、lease、attempt 和 expiry/retry。

## 4. 测试计划

### 4.1 JobStore 与状态

- create Job + accepted event 原子。
- planning -> queued -> running -> terminal。
- 同 idempotency key 同 body 返回原 Job。
- 同 key 不同 body 冲突。
- terminal state 不被迟到回调覆盖。
- Dispatcher/runtime 重启恢复 queued/running/cancelling Job。
- JobView counts/progress 与 unit 状态一致。

### 4.2 Lease 与 retry

- 两个 local slots 竞争同一 unit，只发一个 lease。
- acquire 后不 start，lease 过期重新 ready。
- running lease renew。
- retryable execution/storage/resource 错误重试。
- validation/finance/code 不重试。
- stale complete 被拒绝。
- complete 重发返回 duplicate。
- commit 后 complete 前崩溃可安全重新执行或复用结果。

### 4.3 子进程

- operation 确实在不同 PID 执行。
- timeout 终止。
- cancel grace 后 kill。
- crash/非零 exit 分类。
- workspace 清理。
- 多 slot 不超过配置。
- BLAS/OpenMP 环境变量限制过度订阅。

### 4.4 SDK parity

- sync helper 与 submit + wait 相同 JobSpec/结果。
- DataFrame 与 Snapshot 输入数值一致。
- `serialize_boundaries=true` 与直接 InProcess 结果一致。
- JobResult kind 错误转换被拒绝。
- close/reopen runtime 后结果可再次读取。

### 4.5 故障竞态

- cancel vs complete。
- lease expiry vs complete。
- result committed vs Job cancelled。
- event subscriber 断开重连。
- Storage 短暂失败期间继续 renew lease。
- runtime 进程被 kill 后重启。

### 4.6 PAXG local evidence

- 从 P2 Snapshot 提交 `dataset.window@1`，得到 307 行。
- 提交 x1/x4 feature 和 correlation/permutation operation。
- 结果与 P3 直接 Kernel golden 一致。
- Job/result lineage 可追溯到 PAXG Snapshot digest。

## 5. 验收门禁

1. `StockStat.local()` 可从空目录完成 feature/statistics Job。
2. Local Worker 使用子进程且结果先 commit Storage。
3. Dispatcher 重启不丢 queued/running Job，lease 可恢复。
4. retry、stale complete、duplicate complete 和 cancel race 测试通过。
5. Job event sequence 可断线恢复。
6. 同一 operation 直接 Kernel 与 local Job 数值一致。
7. PAXG 307 行基础研究 Job 完成并有完整 lineage。
8. Dispatcher 进程不加载结果 DataFrame。
9. SDK 不 import Worker runtime 或服务数据库。

## 6. 非目标

- 不开放远程 HTTP Client/Worker 服务。
- 不实现 PostgreSQL Scheduler 并发锁。
- 不实现多 Worker 跨机调度。
- 不实现完整 backtest、batch、grid 或 Monte Carlo。
- 不完成 OIDC、多租户配额和生产 sandbox。
- 不提供旧 `StockStatClient` 兼容 facade。

## 7. 风险与处理

| 风险 | 处理 |
|---|---|
| 本地模式为了方便绕过架构 | e2e 断言 JobStore、lease、子进程和 committed Artifact 均存在 |
| SQLite 锁争用 | 本地限制单 Dispatcher，事务短小，P5 使用 PostgreSQL |
| spawn 序列化隐藏引用 | 子进程只接收 WorkUnit DTO 和 locator/config，不传 live service object |
| complete/取消竞态 | state_version + 单事务 terminal 决策 |
| SDK 过早扩张 | P4 只实现验证闭环所需领域入口 |
