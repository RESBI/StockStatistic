# P5：独立 Worker、HTTP 协议与远程部署

> 前置：[P4 Durable Local Jobs](P4.md)
> 输出：独立 Storage/Dispatcher/Worker 服务和 Remote Session
> 后续：[P6 Backtest Kernel](P6.md)、[P7 Compound Experiments](P7.md)

## 1. 阶段目标

把 P4 的同语义组件拆成独立进程和可分离部署服务。Client 只与 Dispatcher/Storage 通信，Worker 通过 HTTP pull + lease 获取任务并直接读写 Storage。P5 的重点是协议一致性、持久调度、故障恢复和“大数据不经过 Dispatcher”。

阶段闭环：

```text
Remote SDK -> Dispatcher HTTP -> PostgreSQL JobStore
Worker HTTP pull -> Storage HTTP/blob -> result commit -> complete
Remote SDK <- SSE/status/result ref <- Dispatcher
Remote SDK -> Storage -> selected result download
```

## 2. 必需交付物

### 2.1 Dispatcher HTTP API

实现并发布 OpenAPI：

```text
POST /v31/jobs
GET  /v31/jobs/{job_id}
GET  /v31/jobs
POST /v31/jobs/{job_id}/cancel
GET  /v31/jobs/{job_id}/result
GET  /v31/jobs/{job_id}/events
GET  /v31/operations
GET  /v31/operations/{operation}
```

要求：

- `StockStat-Protocol: 3.1`。
- request/trace ID。
- Idempotency-Key。
- HTTP Problem error。
- tenant scope 的最小认证 middleware。
- cursor pagination。
- SSE `Last-Event-ID` 恢复和 cursor expired。

### 2.2 Worker HTTP API

实现：

```text
POST /v31/workers/register
POST /v31/workers/{worker_id}/heartbeat
POST /v31/workers/{worker_id}/leases/acquire
POST /v31/attempts/{attempt_id}/start
POST /v31/leases/{lease_id}/renew
POST /v31/attempts/{attempt_id}/partials
POST /v31/attempts/{attempt_id}/complete
POST /v31/attempts/{attempt_id}/fail
POST /v31/attempts/{attempt_id}/cancelled
```

Worker credential 与普通 Client token 分离。lease token 只存 hash，绑定 worker/session/attempt/expiry。

### 2.3 PostgreSQL JobStore

将 P4 repository contract 运行在 PostgreSQL：

- transactional state transition。
- `FOR UPDATE SKIP LOCKED` 或等价唯一 lease 方案。
- scheduler/expiry scanner 的安全并发。
- 多 API 副本共享 JobStore 的基本能力。
- migration、backup/restore smoke。
- SQLite/PostgreSQL 状态语义 contract tests。

首期仍是一个逻辑调度域，不实现 Dispatcher 级联。

### 2.4 Remote Worker Agent

增强 P4 Agent：

- 长生命周期 HTTP clients 和连接池。
- register/heartbeat/acquire/renew/complete loop。
- session replacement。
- exponential reconnect，不在网络错误时忙等。
- drain 和 graceful shutdown。
- capability manifest digest。
- 多 pool/slot。
- Storage HTTP metadata + HTTPS/file locator。
- cache Top-N digest hints。
- upload session 和 commit。
- diagnostic Artifact。

### 2.5 Remote SDK

实现：

```python
StockStat.connect(dispatcher_url, storage_url, token=...)
```

P4 的公开 API 和 JobHandle 在 Remote Session 保持一致：

- HTTP submit/status/cancel/result。
- SSE events 自动重连。
- result member 按需从 Storage 下载。
- transient network retry 只用于安全幂等请求。
- timeout 区分客户端等待超时和 Job execution timeout。

### 2.6 Scheduler 匹配

实现硬约束：

- operation/version。
- kernel/protocol compatibility。
- resource class/pool。
- security profile。
- memory/temp disk/GPU 声明。
- worker session online/draining/quarantined。

实现首批软评分：

- queue age + priority。
- cache hint affinity。
- zone/label affinity。
- tenant 配额的简单权重。

首期不抢占运行中任务。

### 2.7 数据面约束

增加可测试的不变量：

- Dispatcher API body 大小上限。
- Dispatcher 不代理 Artifact bytes。
- assignment 只有 refs 和小型参数。
- Worker 下载 Snapshot partitions 直接来自 Storage/blob。
- Client 下载结果直接来自 Storage/blob。
- file locator 只对同机可信 Worker。
- digest/size 端到端验证。

### 2.8 部署资产

提供：

- Dispatcher Dockerfile。
- Storage Dockerfile。
- Worker Dockerfile。
- 开发 `docker-compose.v31.yml`：PostgreSQL、MinIO、Storage、Dispatcher、1-2 Worker。
- health/readiness endpoints。
- 配置模板和 secret 环境变量清单。
- CLI 启动命令。

旧 `docker-compose.yml` 在 P9 前保留，不在 P5 覆盖。

### 2.9 可观测性基础

实现：

- 结构化日志字段：trace/job/unit/attempt/worker。
- W3C trace context 传播。
- Prometheus 风格 metrics：submit、planning、queue、execution、lease、retry、storage、active slots。
- operation 和错误码低基数标签规则。
- Admin 所需只读资源的最小 API，不实现 UI。

## 5. 测试计划

### 5.1 协议 contract

- OpenAPI 与 contracts JSON Schema golden。
- 所有 endpoint 的合法/非法 request。
- protocol version missing/unsupported。
- HTTP Problem status/code。
- unknown field 和 operation schema error。
- SSE event/schema/sequence/Last-Event-ID。
- pagination cursor 和 tenant scope。

### 5.2 Worker 和调度

- 注册、自检、heartbeat、session replacement。
- capability 缺失时不分配。
- resource pool/slot 匹配。
- draining 不拿新任务。
- two Workers race only one lease。
- priority + aging。
- cache hint 只影响软评分，不违反硬约束。
- protocol/security mismatch 隔离 Worker。

### 5.3 网络故障

- Worker acquire/renew/complete 响应丢失。
- SSE 断开和恢复。
- Storage download/upload 暂时失败。
- Dispatcher API 副本重启。
- PostgreSQL 短暂不可用。
- MinIO 短暂不可用。
- DNS/connection timeout 分类。
- complete 重发和 stale complete。

### 5.4 进程故障

- Worker Agent 被 kill，lease expiry 后重分配。
- 子进程成功、Agent complete 前被 kill。
- Dispatcher 在 lease transaction 后崩溃。
- Storage commit 后 response 丢失。
- Worker 新 session 替换旧 session。

### 5.5 数据面和性能

- 50 MB/500 MB Snapshot/Artifact 的 bytes 不出现在 Dispatcher access/body instrumentation。
- Dispatcher RSS 不随数据文件体积线性增长。
- Worker 第二次使用相同 Snapshot 命中 cache。
- 多 task 同节点只下载同 digest 一次。
- assignment/complete payload 保持小型。

### 5.6 Local/Remote parity

对相同 JobSpec：

- operation parameters digest 相同。
- 状态序列语义等价。
- Error code 等价。
- final manifest kind/schema 等价。
- P3 数值结果一致。
- PAXG feature/statistics result digest 在可确定编码下相同。

## 6. 验收门禁

1. compose 环境可从空数据库启动并完成远程 Job。
2. Remote SDK 与 Local SDK 使用同一业务调用代码。
3. PostgreSQL 下并发 Worker 不会重复发有效 lease。
4. Dispatcher/Worker/Storage 任一进程重启后任务可恢复或正确失败重试。
5. SSE 可断线续订，事实状态不依赖 SSE。
6. PAXG 基础研究 Job 在单远程 Worker 和 2 Worker 部署结果一致。
7. 50 MB/500 MB 数据不经过 Dispatcher，缓存第二次命中。
8. trusted/signal capability 分池匹配正确。
9. OpenAPI、contracts schema 和 SDK DTO contract tests 全部通过。

## 7. 非目标

- 不实现完整回测能力。
- 不实现 batch/grid/Monte Carlo planner。
- 不实现 Dispatcher 多级级联。
- 不实现 Kubernetes Operator 或云厂商 autoscaler。
- 不完成生产 OIDC/mTLS PKI；先提供可替换认证端口和测试 credential。
- 不实现共享内存跨机协议。

## 8. 风险与处理

| 风险 | 处理 |
|---|---|
| HTTP 与 InProcess 业务逻辑分叉 | 两者只做 adapter，共用 application services 和 contract suite |
| 网络重试导致重复提交 | Idempotency-Key、lease token、digest 和 complete 幂等 |
| Dispatcher 悄然代理数据 | instrumentation 测试和路由层不提供 blob proxy |
| 多副本 scheduler 重复租约 | DB row lock/唯一约束和竞态测试 |
| Worker 自报 capability 不可信 | self-test + manifest digest；P9 加强镜像/签名校验 |
